Voice Call Reliability Audit & Incoming Call System

목표:

현재 구현된 LiveKit 기반 음성통화 기능을 점검하고, 실제 전화처럼 동작하도록 개선합니다.

필수 요구사항:

1. 전화를 거는 사람이 통화 연결 전까지 신호음을 들어야 함
2. 전화를 받는 사람은 벨소리/수신 화면을 받아야 함
3. 앱이 백그라운드 또는 내려간 상태여도 전화 수신 알림이 와야 함
4. 통화 수락/거절/부재중 처리가 되어야 함
5. 기존 채팅/통화 카드/LiveKit 연결 구조는 깨지지 않아야 함

---

1. 현재 구조 점검

먼저 기존 코드를 분석해주세요.

확인할 것:

* LiveKit room 생성 위치
* call card 생성 위치
* call 상태 관리 방식
* 발신자/수신자 구분 방식
* 채팅방 내 통화 표시 방식
* iOS 오디오 세션 설정
* push notification 설정 여부
* 앱 백그라운드 상태에서 수신 처리 가능 여부
* 통화 종료/실패 처리 방식

아직 수정하지 말고 분석 후 필요한 변경 파일 목록을 먼저 정리해주세요.

---

2. 통화 상태 모델 정리

통화에는 아래 상태가 필요합니다.

call_status:

* ringing
* accepted
* declined
* missed
* cancelled
* ended
* failed

발신자 흐름:

1. 전화 걸기
2. call_status = ringing
3. 발신자에게 신호음 재생
4. 상대가 수락하면 accepted
5. LiveKit 연결
6. 통화 종료 시 ended

수신자 흐름:

1. incoming call 수신
2. 벨소리 재생
3. 수신 화면 표시
4. 수락 시 accepted
5. 거절 시 declined
6. 일정 시간 무응답 시 missed

---

3. DB / API 보강

필요하면 calls 또는 call_sessions 테이블을 추가/확장해주세요.

필드 예시:

* id
* room_id
* caller_user_id
* callee_user_id
* chat_room_id
* livekit_room_name
* status
* started_at
* accepted_at
* ended_at
* declined_at
* missed_at
* cancelled_at
* created_at
* updated_at

API 예시:

POST /calls

* 전화 걸기
* call session 생성
* 상대에게 push notification 발송

GET /calls/incoming

* 현재 수신 대기 중인 전화 확인

POST /calls/:id/accept

* 전화 수락
* LiveKit token 발급

POST /calls/:id/decline

* 전화 거절

POST /calls/:id/cancel

* 발신자가 전화 취소

POST /calls/:id/end

* 통화 종료

4. 발신자 신호음

전화를 건 사람은 status=ringing 동안 신호음을 들어야 합니다.

구현:

* 모바일 앱에 outgoing ringtone/ringback tone 추가
* 상대가 수락/거절/부재중/취소되면 즉시 중지
* 앱이 foreground일 때는 로컬 오디오 재생
* 백그라운드에서는 OS 정책상 제한이 있으므로 foreground 중심 구현

주의:

* 신호음이 중복 재생되지 않게 처리
* 통화 화면을 벗어나도 status가 바뀌면 멈추게 처리

---

5. 수신자 벨소리 / 수신 화면

앱 foreground:

* incoming call modal 표시
* 벨소리 재생
* 수락/거절 버튼

앱 background / killed 상태:

* push notification 발송
* 알림 클릭 시 통화 수신 화면 또는 call detail로 이동

iOS:

* 진짜 전화처럼 보이게 하려면 CallKit 도입 검토
* Expo Managed 환경이면 CallKit/VoIP Push에 제약이 있을 수 있음
* 현재 환경에서 가능한 범위를 확인하고, 최소 구현은 push notification + foreground incoming modal로 처리

Android:

* FCM high priority notification
* 가능하면 full-screen intent 또는 heads-up notification 검토
* Expo 환경 제약 확인

---

6. 백그라운드 수신

중요:

일반 push notification만으로는 iOS에서 완전한 전화 수신 경험이 제한될 수 있습니다.

점검할 것:

* 현재 앱이 Expo Managed인지, Dev Client인지, Bare인지
* expo-notifications 사용 여부
* FCM/APNs 설정 여부
* iOS CallKit 사용 가능 여부
* Android foreground service/full-screen notification 사용 가능 여부

결론을 코드 주석 또는 문서로 정리해주세요.

MVP 구현 기준:

* foreground: 즉시 벨소리 + 수신 모달
* background: push notification
* notification tap: 통화 화면 진입
* missed timeout 처리

---

7. Push Notification

상대방에게 전화가 오면 push notification을 보내야 합니다.

알림 내용:

title:
"전화가 왔습니다"

body:
"{callerName}님이 음성통화를 요청했습니다."

data:
{
"type": "incoming_call",
"callId": "...",
"chatRoomId": "...",
"callerUserId": "..."
}

주의:

* 수신자가 차단한 사용자면 전화 걸기 차단
* 친구가 아닌 사용자에게 전화 가능한지 현재 정책 확인
* 필요하면 친구 또는 같은 채팅방 참여자만 전화 가능

---

8. 통화 타임아웃

ringing 상태가 일정 시간 지속되면 자동 missed 처리

권장:

* 30초 또는 45초

발신자:

* “응답이 없습니다.”

수신자:

* 부재중 전화 기록

구현:

* 서버에서 timeout 처리하는 방식 우선
* 어려우면 API 조회 시 만료된 ringing call을 missed로 정리
* 프론트 타이머만 믿지 말 것

---

9. 통화 카드 / 채팅 기록

채팅방에는 통화 결과 카드가 남아야 합니다.

예시:

* 음성통화 요청
* 통화 종료 03:21
* 부재중 전화
* 통화 거절
* 통화 취소

기존 통화 카드가 있다면 상태 값을 연결해주세요.

10. 테스트

확인할 것:

1. A가 B에게 전화 걸면 call_status=ringing
2. A는 신호음을 듣는다
3. B가 foreground면 수신 모달 + 벨소리
4. B가 수락하면 양쪽 LiveKit 연결
5. B가 거절하면 A 신호음 중지 + declined 표시
6. A가 취소하면 B 벨소리 중지 + cancelled 표시
7. 30~45초 무응답 시 missed 처리
8. 앱 background 상태에서 push notification 수신
9. 알림 클릭 시 통화 화면으로 이동
10. 통화 종료 시 ended 기록
11. 중복 수락/중복 종료 방지
12. 차단 사용자에게 전화 불가
13. 기존 채팅/LiveKit 기능 영향 없음
